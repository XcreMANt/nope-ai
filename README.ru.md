# NOPE

**NOPE — No Opportunistic Production Edits**

NOPE — это система управления AI-assisted разработкой для существующих production-проектов с высокой ценой ошибки.

Система делает акцент не на максимальной автономности или скорости генерации кода, а на контролируемости изменений и снижении риска побочных эффектов.

NOPE особенно полезен для:

- legacy-монолитов;
- изменений в критичном коде;
- проектов с высокой ценой ошибки;
- долгоживущих систем со сложной архитектурой;

NOPE не пытается полностью заменить инженера или принимать решения за него.

Система исходит из того, что AI по своей природе вероятностен и может ошибаться, особенно в больших проектах со сложной логикой и неявными зависимостями.

AI-модели склонны расширять область изменений, делать неподтверждённые предположения и изменять связанные части системы без полного понимания архитектурных и бизнес-последствий таких изменений.

Задача NOPE — уменьшить риск подобных изменений и не допускать ситуации, в которой AI самостоятельно принимает критичные архитектурные или production-решения без контроля со стороны оператора.

---

# Главная идея

Многие AI-инструменты разработки ориентированы прежде всего на скорость выполнения задачи и автономность изменений.

NOPE не пытается писать код как можно раньше, а пытается сначала понять:

- как устроен проект;
- какие части системы затрагиваются;
- какие есть риски и неизвестные;
- что невозможно проверить локально;
- где могут возникнуть побочные эффекты.

---

# Основные принципы

- изменения с учётом рисков проекта;
- минимально необходимый объём изменений;
- запрет “попутного” рефакторинга;
- запрет неподтверждённых предположений;
- контроль изменений со стороны оператора;
- критический обзор рискованных изменений;
- явная фиксация неизвестных и сомнительных мест;
- учёт возможности отката изменений;
- защита критичных пользовательских сценариев.

---

## Ограничения системы

NOPE не гарантирует безопасность изменений и не исключает ошибки AI-моделей.

Система предназначена для снижения риска AI-assisted разработки, а не для полностью автономного изменения production-систем.

NOPE не рассчитан на unattended production changes.

Оператор должен оставаться ответственным за подтверждение рискованных действий и итоговое состояние системы.

# Установка

NOPE рекомендуется устанавливать как отдельный вложенный git repository. Это упрощает обновление системы и отделяет NOPE от кода проекта.

Пример для Codex:

```bash
git clone https://github.com/XcreMANt/nope-ai.git /path/to/project/.codex
```

## Важно

NOPE не является immutable framework.

После adaptation workflow система адаптируется под конкретный проект и начинает хранить project-specific operational knowledge. После этого содержимое `docs/` становится проектной памятью системы.

Перед обновлением рекомендуется проверять возможные конфликты между новой версией NOPE и project-specific documentation.

## После установки

Открыть проект в AI assistant и запустить adaptation workflow.

---

# Adaptation workflow

Перед разработкой необходимо выполнить adaptation workflow:

- изучает проект;
- определяет критичные пользовательские сценарии;
- определяет зоны риска;
- определяет опасные операции;
- определяет ограничения проверки изменений;
- строит рабочую модель проекта.

До завершения adaptation workflow изменять production-код не рекомендуется.

---

# Статус проекта

Разработка проекта находится в стадии переменной активности.

Архитектура, workflow и playbook-ы продолжают калиброваться на реальных production-сценариях.
